# Developer Guide

## Правила проекта

1. Конфигурационный файл должен быть либо полностью загружен, либо программа завершается.

2. Undefined Behavior (UB) недопустимо.
   Любой пользовательский ввод должен быть проверен и валидирован.

3. Исключения используются только для фатальных ошибок.

4. Все ресурсы должны управляться через RAII.

5. Алгоритмы не выполняют ввод-вывод.

6. Interface отвечает только за чтение команд и отображение сообщений.

7. Control управляет состоянием программы и вызывает алгоритмы.

8. Алгоритмы не должны знать о существовании Interface.

9. Любой параметр, считанный из файла, должен быть валидирован.

10. Все имена переменных должны быть понятными и отражать их назначение.

11. После добавления новой команды или функции изменения должны быть
    отправлены в отдельную ветку и оформлены через Pull Request.
    Небольшие изменения проще проверять и сопровождать.


## Как добавить новую команду

### Документация
1. Добавить описание команды в User Guide и туда же описание знач по умолчанию.
2. Добавить параметры по умолчанию в config.conf.
3. Добавить команду в командные файлы

### Интерфейс
4. Добавить команду в help и в вывод консоли.
5. Добавить чтение параметров из конфигурационного файла + добавить команду в список команд в конфиге
6. Добавить новые поля в DispatcherParams.
7. Реализовать чтение параметров в Interface.
8. Добавить проверку параметров в Validator.

### Бизнес-логика
9. Передать параметры в Control.
10. Обработать команду в Control.
11. Реализовать алгоритм либо создать новый класс.

### Сборка
12. Если был создан новый *.cpp-файл, добавить его в соответствующий CMakeLists.txt.


## Как работать с бенчмарком

0. Лучше первым запуском протестировать, включите логи, картинки, а в бенчмарке пусть цикл пройдет 1-2 раза
1. В конфиге отключите логи. Нужно поставить режим OFF
2. В конфиге выбирите режим Random для seed, чтобы карты были разные при новом запуске
3. Напишите командный файл, где укажите все параметры алгоритмов. Нельзя использовать значения по умолчанию (Чтобы вся информация о среде была в командном файле)
4. Используйте команду g_auto в командном файле со всеми ее параметрами для генерации случайных карт
5. Нельзя использовать визуализацию в командном файле (Время работы станет очень большим, а картинки просто будут перезаписываться)
6. Добавьте команду save_metrics и укажите ее уникальный параметр, чтобы другие командные файлы случайно не написали в csv таблицу чужие метрики
7. В скрипте benchmark.sh напишите имя своего командного файла
8. Запускаем скрипт


## Работа с гитхабом
##### ======================================
## 🔄 SAFE WORKFLOW v1.0
##### ======================================
#### Философия: "Мои изменения — священны 😤️, main обновляется без боли 😇️"

### 1️⃣ Начало работы (обновляем локальный main удаленным)
```
git checkout main
git pull origin main
```

### 2️⃣ Создаем ветку для задачи
```
git checkout -b test # Создаем от актуального main
```

### 3️⃣ Работаем в ветке
```
[вносим изменения в файлы]
git status # Проверяем изменения
```

### 4️⃣ Фиксируем изменения
```
git add .
git commit -m "feat: добавил функционал X"
git push origin test
```

### 5️⃣ Мержим с main (через PR)

### 6️⃣️ Финализируем main
```
git checkout main
git pull origin main # Локальное обновление
```

### 7️⃣ Создаем тег (релиз)
```
git tag -a v29.0.0 -m "Изменения:
1) Улучшил визуализацию
2) Исправил скрипт, теперь работает как задумывалось"
git push origin v29.0.0
```

### 8️⃣ Удаляем ветку
```
git branch -d test
git push origin --delete test
```

### 🔥 Важно!
```
- Все мержи в main ТОЛЬКО через Pull Request
- Коммиты в ветке должны быть атомарными
- Названия веток по шаблону: `feature/...`, `fix/...`, `hotfix/...`
```
